Impuls
Einstieg · Die Wunde

„Wo ist mein Code hin?"

  • Agent A ändert eine Datei — committet nicht.
  • Agent B liest dieselbe Datei, sieht die Änderung nicht, baut auf dem alten Stand weiter.
  • Zusammenführen → Konflikt, Doppelarbeit, teils verlorene Arbeit.
Das ist kein Tooling-Bug. Es ist die Grundmechanik agentischer Systeme — und der Schlüssel zu allem, was folgt.
Impuls
Einstieg · Was gerade passierte

Jeder Agent hat sein eigenes Gedächtnis

  • Agent B war nicht „dumm" — er konnte A's Arbeit nicht sehen. Sie stand nur in A's Kontext (und uncommittet auf der Platte).
  • Kontext ist pro-Agent isoliert.
  • Was nicht committet ist, existiert für andere nicht.
Fast jede Regel unseres Systems lässt sich aus diesen zwei Sätzen herleiten — nicht auswendig lernen.
Impuls
Mentales Modell

Agent = LLM + Werkzeuge + Schleife

  • Ein LLM, das Werkzeuge aufrufen darf — Dateien lesen/schreiben, Shell, Web —
  • und in einer Schleife arbeitet, bis die Aufgabe erledigt ist.
  • Chatbot antwortet. Agent handelt.
Weil ein Agent handelt, muss er kontrolliert werden wie ein Mitarbeiter — nicht genutzt wie eine Suchmaschine.
Impuls
Mentales Modell · Kernkonzept

Subagenten & Kontext-Isolation

  • Main-Agent koordiniert.
  • Jeder Subagent: eigener Kontext + eigene Werkzeuge + eine Rolle.
  • Subagenten sehen weder einander noch den vollen Verlauf.
Mentales Modell

Isolation ist Feature und Fluch

Feature

  • Fokus, kein Rauschen
  • Parallelisierbar
  • Begrenzte Fehlerfläche

Fluch

  • Blind für die Arbeit der anderen
  • „Vergisst" über Sessions
  • Braucht explizite Übergabe
Aus dem Fluch folgen direkt: schriftliche Reports, das Commit-Gate und persistentes Memory.

Impuls
Das System

Fünf Pfeiler

  1. Verfassung — CLAUDE.md
  2. Team — Subagenten mit eigenen Rechten
  3. Gedächtnis — persistentes Memory
  4. Werkzeuge — MCP & Skills
  5. Nebenläufigkeit — geteilter State, mehrere Sessions
1–4 bauen das System. 5 hält es zusammen, sobald mehr als eine Session läuft.
Pfeiler 1 · Verfassung

CLAUDE.md — die Projektverfassung

  • Wird jede Session geladen. Regeln, die Standardverhalten überschreiben.
  • Routing-Matrix · Code-Regeln (User-Isolation, kein console.*) · Worktree-Pflicht · Shell-Patterns.
Es ist ein Prompt, kein Wiki. Faustregel: Muss sich die Regel ändern, sobald sich der Code ändert? → dann gehört sie in den Code.
Pfeiler 2 · Team

Spezialisten mit eigenen Rechten

  • ~18 Spezialisten: frontend, services, backend, security, compliance, devops, architect, quality …
  • Jeder hat eingeschränkte Werkzeuge.
  • Rechte = Sicherheit: kein Werkzeug → kein Schaden.
Pfeiler 3 · Gedächtnis

Was Sessions überdauert

  • Auto-Memory: Index MEMORY.md + eine Datei pro Fakt (Frontmatter, verlinkt).
  • agent-memory: pro Agent eine eigene Wissensbasis.
  • Aktiv kuratiert — falsche Memories werden gelöscht, nicht angehäuft. Kein Log.
Memory-Dateien committen — sonst sehen parallele Sessions die Lektion nie. (Kommt in Pfeiler 5 wieder.)
Pfeiler 4 · Werkzeuge

Zugriff auf die echte Welt

  • MCP-Server — standardisierte Schnittstelle zu externen Systemen: Wiki, DB-Backend, Browser-DevTools, Zahlungsanbieter-Anbindung.
  • Skills — paketierte, wiederverwendbare Abläufe.
  • Hooks — das Werkzeug, mit dem wir Regeln erzwingen (nächster Block).
Werkzeuge sind Fähigkeit und Angriffsfläche. Nur geben, was die Rolle wirklich braucht.
Impuls
Roter Faden

Weiche vs. harte Durchsetzung

  • CLAUDE.md-Regel = Bitte. Das Modell sollte folgen — probabilistisch.
  • Hook = Gesetz. Die Harness erzwingt — deterministisch.
Die Kunst: nur die wenigen katastrophalen Regeln hart machen.
Roter Faden · Hooks

Hooks — Gesetze auf Werkzeug-Ebene

  • Kennt ihr von git: pre-commit blockt einen Commit.
  • Claude-Code-Hooks blocken/prüfen Tool-Aufrufe: PreToolUse, PostToolUse …
  • Unser Beispiel: block-main-edits.sh blockt Edits an src/ auf main.
git-hooks hängen an git-Events, Claude-Code-Hooks an Tool-Events — jedem Datei-Edit, jedem Bash-Aufruf. Die einzige deterministische Kontrolle im System.
Roter Faden · Design-Prinzip

Welche Regel macht man hart?

  • Achse Schaden × Verletzungswahrscheinlichkeit.
  • Nur hoch × hoch → Hook.
  • Rest → Leitplanke in CLAUDE.md.
~30 Regeln, nur eine Handvoll Hooks. Mechanik ist teuer — sparsam einsetzen.
Prozess

Wie eine Aufgabe wirklich durchläuft

Nichts wird direkt auf main editiert. Nichts geht ungeprüft raus.
Prozess · Handoff

Die Übergabekette

Jede Übergabe ist explizit und schriftlich — direkte Folge der Kontext-Isolation. Der nächste Agent sieht nur, was als Datei vorliegt.
Prozess · Gates

Gates als Immunantwort

  • Pre-Edit-Gate · Commit-Gate · Post-Agent-Gate · Review-Artifact-Regel.
  • Ein Mensch, der User-Isolation vergisst, fällt im Review auf.
  • Ein Agent produziert 200 Zeilen konsistenten Code mit demselben Fehler — der nächste baut darauf auf.
Die Gates sind die Ersatz-Immunantwort für ein System, dem das menschliche „Moment mal…"-Bauchgefühl fehlt. Keine Bürokratie — Notwendigkeit.

Pfeiler 5 · Nebenläufigkeit

Deine Dev-Umgebung ist ein verteiltes System

  • Ein Dev, eine Session → alles einfach.
  • 4 Devs × 2 Sessions auf einem Repo, einer Staging-DB, einer CI → Race Conditions.
Ihr kennt dieses Problem. Es trifft euch nur auf einer neuen Ebene.
Nebenläufigkeit · Demo

Zwei VS-Code-Fenster, dasselbe Repo

  • A editiert, committet nicht. B liest den alten Stand, baut weiter.
  • Zusammenführen → Kollision.
Derselbe Mechanismus wie im Opener — nur zwischen Sessions statt zwischen Agenten.

Nebenläufigkeit · Echte Vorfälle

Drei, die uns wirklich getroffen haben

Stille Korruption

INSERT IGNORE → ~20 DB-Objekte kaputt

File N+5 korrigiert File N — IGNORE verwarf die Korrektur schweigend. „Grün, keine Fehler, trotzdem kaputt."

Fix: ON DUPLICATE KEY UPDATE
Falsche Sicherheit

Dev-Server servierte 2× den falschen Worktree

Geteilter Prozess an fremden Branch gepinnt (CWD nicht-deterministisch) — Briefing behauptete fälschlich „confirmed".

Die Kollision lügt dich an

Nummern N-b / N-a…d + Datei-Drift

Hands-on

Baut euer eigenes Mini-System

  • Wegwerf-Repo, neutrale App. Ein Vertreter jedes Pfeilers.
  • Wenn das läuft, habt ihr das Muster — skalieren kommt später.
  • Setup: VS Code + Claude Code, ein leeres Repo.
Nicht die Breite lernen, sondern den minimalen selbsttragenden Kern. Verstehen durch Nachbauen im Kleinen.
Hands-on · Die 5 Zutaten

Fünf Dateien, ein System

  1. CLAUDE.md — mit 3 Regeln
  2. Zwei Subagenten — implementer + reviewer, reviewer ohne Write
  3. Ein Gate als Hook — blockiere Edits auf main
  4. Eine Memory-Datei
  5. Ein Worktree-Flow
Zutat 3 ist zugleich ein Nebenläufigkeitsschutz. Zutat 2 zeigt „Rechte = Sicherheit" am eigenen Leib. Schritt für Schritt im Kit → hands-on-kit/README.md.
Abschluss · Skalieren

Vom Mini zum echten System

  • Wachst entlang des Schmerzes, nicht auf Vorrat. Jede Regel entsteht aus einem konkreten Vorfall.
  • Reihenfolge: erst Routing + Worktrees → dann Hooks für das Katastrophale → dann Memory → zuletzt Nebenläufigkeits-Schutz, wenn das Team wächst.
Nicht mit 18 Agenten und 30 Regeln starten. Unser System ist über Monate reaktiv gewachsen.
Abschluss · Ehrlichkeit

Was wir noch nicht gelöst haben

  • Geteilte Staging-Datenbank — mehrere Sessions, eine DB.
  • CI-Runner-Contention — begrenzte, geteilte Runner.
  • stale-base-Plan-Gate-Deadlocks — Branch hinter main.
  • Build-Datei-Drift (z. B. Dockerfile-COPY) — zwei gekoppelte Dateien, keine CI-Prüfung (3× aufgetreten).
Ehrlichkeit > geglättete Erfolgsstory. Ihr könnt unsere offenen Baustellen von Anfang an vermeiden.
